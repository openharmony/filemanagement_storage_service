# OH-CLI Creator

将 HarmonyOS/OHOS 仓库中的系统能力封装成可执行 CLI 工具，并产出完整的接口映射与编译验证报告。

## 使用场景

- 给 HarmonyOS/OHOS 组件新增 CLI 工具
- 把 SA、`inner_kits`、`inner_api/native`、IDL/proxy 能力封装成命令行工具
- 需要输出"哪些接口已封装、哪些未封装、为什么这样设计"的完整报告
- 需要符合 `ohos-`/`hms-` 命名规范的 CLI 开发
- 需要记录从接口探索到编译验证的全过程文档

## 分层工作流

本技能按以下**5 个阶段**执行，**每个阶段完成后需等待用户确认**才能进入下一阶段：

### 通用行为规范（适用于所有阶段）

每个阶段的输出完成后，按以下流程与用户交互：

1. **展示阶段输出**：将本阶段的分析结果、设计方案或实现产物呈现给用户
2. **等待用户反馈**：用户可能直接确认，也可能提出修改意见或质疑
3. **处理用户反馈**：
   - 如果用户**无异议** → 直接进入下一阶段
   - 如果用户**提出修改意见** → 立即根据意见调整，**重新生成完整的阶段总结**，展示给用户确认
   - 重复步骤 3，直到用户明确表示无异议
4. **用户明确确认后** → 才可进入下一阶段，**严禁在用户未确认的情况下擅自推进**

## 命名规则（必读）

根据仓库路径确定前缀：
- **开源仓**（`/foundation/`、`/base/`、`/applications/` 等）：使用 `ohos-` 前缀
- **闭源仓**（`/vendor/huawei/`）：使用 `hms-` 前缀

**目录结构**：
```
tools/
  <cli-name>/                  # 如：ohos-network-scanner/ 或 hms-device-auth/
    ├── src/
    │   └── main.cpp          # 主入口文件
    ├── tests/                # 测试代码
    ├── docs/                 # 文档目录
    │   ├── README.md
    │   └── USAGE.md
    ├── BUILD.gn              # 构建配置
    ├── config.json           # 工具描述文件
    └── <cli-name>_report.md  # 封装报告
```

**关键规则**：
1. **只使用小写字母和连字符（`-`）**，禁止下划线（`_`）
2. **功能领域 - 工具类型结构**（如 `network-scanner`、`package-manager`、`device-auth`）
3. **禁止缩写**（使用完整单词，避免 `mgr`、`cfg`、`vol` 等）
4. **长度限制**：前缀后的名称部分不能超过 16 个字符
5. BUILD.gn 中 target 名称与目录名一致

## 实现约束

1. **禁止空实现**：每个命令必须调用真实系统接口
2. **错误码对齐**：必须使用源码实际错误码（如 `E_OK = 27787264`）
3. **静态命令表**：使用 `std::unordered_map` 分发，禁止长串 `if-else`
4. **LRO 模式**：耗时>5 秒的任务使用异步提交 + 轮询模式
5. **输出格式**：stdout 输出 JSON，stderr 输出日志
